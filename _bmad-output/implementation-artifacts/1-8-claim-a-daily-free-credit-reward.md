# Story 1.8: Claim a daily free-credit reward

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a returning player,
I want a daily reward I can claim,
So that I get a small free top-up for coming back.

## Acceptance Criteria

**AC-1 — First claim of the day grants the configured amount and records the claim**
**Given** a player who has not claimed today (per `IClock` calendar day, UTC)
**When** they claim the daily reward
**Then** the Credit balance increases by the configured daily-reward amount (read from `EconomyConfig`, AR-16)
**And** `SaveModel.DailyClaim` is recorded as today's UTC date (ISO-8601 `yyyy-MM-dd`)
**And** the balance delta AND the `DailyClaim` write are committed in ONE atomic save (AR-8).
[Source: docs/epics.md#Story 1.8; docs/prds/prd-Veilwalkers-2026-06-10/prd.md#FR-2; docs/architecture.md#AR-16, AR-18; Assets/Veilwalkers/Persistence/SaveModel.cs:81-85]

**AC-2 — A second claim the same calendar day is refused, balance unchanged**
**Given** a player who already claimed today (`DailyClaim` equals today's UTC date)
**When** they attempt to claim again the same calendar day
**Then** the claim is refused via a typed result (no throw), the balance is unchanged, and `DailyClaim` is unchanged.
[Source: docs/epics.md#Story 1.8; docs/prds/…/prd.md#FR-2 ("at most once per calendar day"); docs/architecture.md#AR-7 (typed-result, never throw for an expected failure)]

**AC-3 — The reward becomes claimable again after the UTC day rolls over; no logic reads `DateTime.Now`**
**Given** the clock seam (`IClock` injected; a fake clock controls "now" in tests)
**When** the UTC day rolls over (the fake clock advances past midnight UTC)
**Then** the reward becomes claimable again and a fresh claim succeeds (balance increases, `DailyClaim` updates to the new date)
**And** NO production logic reads `DateTime.Now`/`DateTime.UtcNow`/`Time.time` directly — the current day is derived only through `IClock.UtcNow` (AR-18). (This is enforced behaviorally: the day-rollover and boundary tests only pass if the service's clock is fully fakeable; there is no static analyzer guarding the call, so the test coverage IS the guarantee.)
[Source: docs/epics.md#Story 1.8; docs/architecture.md#AR-18; Assets/Veilwalkers/Core/IClock.cs:5-16]

**AC-4 — The daily-reward amount is data-driven (EconomyConfig), not hard-coded**
**Given** `EconomyConfig` holds the tunable economy numbers (Story 1.6, AR-16)
**When** the daily reward is granted
**Then** the granted amount is read from a new `EconomyConfig.DailyRewardCredits` accessor (a `[SerializeField]` field, OQ-9-provisional like the rest of the non-canon numbers)
**And** the amount is NOT a constant in `DailyRewardService` logic — changing the asset value changes the reward with no service recompile.
[Source: docs/architecture.md#AR-16; Assets/Veilwalkers/Economy/EconomyConfig.cs:11-58 (the data-driven contract + the existing provisional fields)]

> **AC-4 is the AR-16 application of the epic's "by the configured amount" wording.** Unlike the Story 1.7 starting grant — a fixed onboarding constant deliberately kept OUT of `EconomyConfig` (deviation #3 there) — the daily reward IS a balancing knob (OQ-9 will tune "how much should returning players get"), so it belongs in `EconomyConfig` alongside the other tunables. This is the inverse decision from 1.7 and is intentional.

### The load-bearing design decision: ONE atomic write, not two

The single most important correctness choice in this story is that the claim is **one lock-held mutation of two fields** (`Credits` += amount AND `DailyClaim` = today), persisted once — NOT a `GrantCreditsAsync` call followed by a separate `DailyClaim` write.

Why this matters: Story 1.7's `FirstLaunchGrant` legitimately uses two writes (grant, then marker) because a crash between them is *benign* — it re-grants on the next launch, which is exactly the accepted reinstall behavior. **The daily reward's inter-write crash is NOT benign:** if credits were granted but `DailyClaim` did not persist, the retry would grant AGAIN the same day, violating AC-2 ("already claimed today → refused, balance unchanged"). The only way to guarantee the credit delta and the claim-date move together-or-not-at-all is to mutate both in one in-memory step and persist once, rolling BOTH back on failure. This is precisely what AR-8 mandates ("one atomic save write per player action … roll back the whole in-memory mutation on persist failure") and why the story must not decompose the claim into two persists.

### Sanctioned-deviation matrix (check before classifying any review finding)

1. **`DailyRewardService` does its OWN single lock-held two-field write; it does NOT call `ICreditService.GrantCreditsAsync`.** A reviewer asking to "reuse the grant primitive" is asking to split one atomic action into two persists with an unsafe inter-write window (a same-day double-claim on a crash — an AC-2 violation). The single-write design IS the AR-8-correct choice. (Contrast 1.7, where two writes were safe because the crash outcome was a benign re-grant.) Both fields roll back together on any persist failure.
2. **The daily-reward amount lives in `EconomyConfig` (`DailyRewardCredits`), unlike 1.7's `StartingCredits` constant.** This is the deliberate inverse of 1.7 deviation #3. The starting grant is a fixed onboarding constant (not a balancing knob); the daily reward IS a balancing knob (OQ-9 will tune it), so AR-16 puts it in the config. A reviewer citing "1.7 kept its amount out of EconomyConfig, so 1.8 should too" has the rationale backwards — the test is "is this a number a designer balances?", and the daily reward is.
3. **Calendar day = UTC, derived from `IClock.UtcNow`, compared as a `yyyy-MM-dd` string.** No doc specifies device-local; every time reference in the architecture is UTC (`IClock` doc, `SaveModel.DailyClaim` doc "ISO-8601 UTC date"). Keying off the UTC date (not the device-local date) is correct and is pinned by a dedicated test. A reviewer wanting local-day semantics is contradicting the AR-18/`IClock` UTC contract; if product ever wants local-day, that is a future change with its own decision, not a bug here.
4. **`DailyRewardService` is constructor-injected `IClock`; it does not read `GameServices.Get<IClock>()`.** Pure-logic Economy services receive deps by ctor injection (AR-4: "only MonoBehaviours/UI read the locator"). ### Review Findings

Code review 2026-06-13 (3 adversarial layers via `story-cr-fanout`; 12 raw findings → verify-disputed-first triage: **4 patch, 4 defer, 3 spec-sanctioned, 1 false-positive**). The review found one real correctness gap (the future-date guard) plus two contract-tightening fixes and a coverage gap; the single-atomic-write design and the sanctioned deviations all held up.

- [x] [Review][Patch] **`IsClaimable` admitted future-dated stamps (MAJOR)** [Assets/Veilwalkers/Economy/DailyRewardService.cs] — RESOLVED. The guard was `storedClaim != today`, which treats a stored date AHEAD of today (tampered/corrupt save, or a clock that jumped backward) as claimable — re-opening the same-day double-claim AC-2 forbids once the real day catches up. Changed to `string.CompareOrdinal(storedClaim, today) < 0` (claimable iff never-claimed OR strictly earlier; `yyyy-MM-dd` lexicographic order == date order). New regression test `A_future_dated_stored_claim_is_refused_not_treated_as_claimable` pins it.
- [x] [Review][Patch] **`checked` overflow was a third, undocumented throw path on `TryClaimAsync`** [Assets/Veilwalkers/Economy/DailyRewardService.cs] — RESOLVED. The `checked { Credits += amount }` block sat OUTSIDE the inner try/catch, so an `OverflowException` (pathological near-`int.MaxValue` balance) would escape as a faulted task — violating the seam's "throws only the before-load `InvalidOperationException`, every other failure is a typed result" contract. Moved the arithmetic INSIDE the inner try, so overflow is caught and converted to a `PersistenceFailed` result with both fields rolled back. (Existing persist-failure test covers the catch path; the log message generalized from "did not persist" to "did not complete" and the test regex updated to match.)
- [x] [Review][Patch] **Recovery-swap reported a stale `priorBalance` as `NewBalance`** [Assets/Veilwalkers/Economy/DailyRewardService.cs] — RESOLVED. After a recovery swap, `_saveService.Current` is a DIFFERENT model than the detached `model`, so `priorBalance` could disagree with the live balance that `DailyRewardResult.NewBalance` is documented to carry ("the unchanged current balance"). Changed the swap branch to report `_saveService.Current?.Credits ?? priorBalance` (the authoritative balance, falling back only if recovery also nulled `Current`). The `OnDailyRewardClaimed` event already documents staleness, but the result value now matches its own doc.
