# Story 3.3: Own the AR session lifecycle with prewarm

Status: done

## Story

As a player,
I want AR to be ready the moment I commit to hunting,
So that the first-session magic moment isn't lost to loading.

## Acceptance Criteria

**AC-1 (sole owner of the AR lifecycle; the cold→prewarming→ready→running→paused state machine).**
**Given** `ArSessionService` as the sole owner of the AR lifecycle
**When** the session runs
**Then** it transitions cold → prewarming → ready → running → paused
**And** it is the ONLY place AR is started or torn down (no hand-rolled session anywhere else).
[Source: docs/epics.md#Story-3.3; docs/architecture.md:405 (named home), :466-468 (AR boundary — sole owner), :584-587 (warmup ownership rule); AR-14; FR-4 staging]

**AC-2 (staged cold-start; AR warmup + preload are on the async path and NOT awaited before Home is interactive).**
**Given** the staged cold-start budget
**When** the app boots (`Bootstrap.LoadPhase { EssentialSync, WarmupAsync, Ready }`)
**Then** essential services wire synchronously while AR warmup + first-Monster preload run on the async path
**And** AR warmup + preload are NOT awaited before Home is interactive (the AC-CS-2 regression guard — the contract CI asserts, not a millisecond budget).
[Source: docs/epics.md#Story-3.3 (AC-CS-2); docs/architecture.md:577-597 (NFR-5 staged-loading budget, CI-vs-device); AR-14]

**AC-3 (prewarm is triggered by Onboarding/Home, not owned by them; cheap tear-back; backgrounding / permission-race / double-start all handled inside the service).**
**Given** the AR-entry affordance is visible/likely
**When** prewarm is triggered (Onboarding/Home call `ArSessionService.PrewarmAsync()` — they do NOT warm the session themselves)
**Then** the session can cheaply tear back to cold if the player backs out
**And** backgrounding, permission-race, and double-start are all handled inside `ArSessionService` (the trigger lives outside the owner; the lifecycle stays inside it).
[Source: docs/epics.md#Story-3.3; docs/architecture.md:583-590 (warmup ownership rule — trigger outside, lifecycle inside; gate + cheap tear-back); deferred-work.md (the 3.1 CR deferral owned by 3.3 — re-validate camera permission before session start)]

### The design decisions this story locks (call them out for the CR reviewer)
1. **`ArSessionService` is the pure-logic state machine; `IArSession` is the OS-touching adapter — the lifecycle DECISIONS live in the service, the ARCore glue lives in the adapter.** This reconciles the architecture's "AR contains no branching logic worth testing — thin adapter" mandate (architecture.md:466) with the "sole owner of the AR lifecycle" requirement (architecture.md:405, :584): the *adapter* (`ArcoreSession`/`IArSession`) holds no branching (it is the thin glue the mandate targets), while the cold→prewarming→ready→running→paused state machine (branching genuinely worth testing) lives in the pure-logic `ArSessionService`, headless-tested against `FakeArSession`. Identical reconciliation to 3.1's `CameraPermissionFlow`/`ICameraPermission` and 3.2's `ArSafetyGate`. **This is the canonical "no `ARSession` in EditMode → seam the adapter, test the state machine" pattern (architecture.md:591-592).**
2. **The lifecycle is async at exactly one seam (`StartAsync`) and double-start-safe.** ARCore session start is not instantaneous, so `IArSession.StartAsync` is async and `PrewarmAsync` awaits it. Everything else (pause/resume/stop) is a cheap synchronous toggle. `PrewarmAsync` is idempotent: a second call while `StartAsync` is in flight must NOT fire a second start (a single in-flight `Task`/state guard) — this is the architecture's "double-start handled inside `ArSessionService`" (architecture.md:585). Pinned by the in-flight `StartCount == 1` test.
3. **Prewarm is an OPTIMIZATION, not a precondition.** `EnterAsync` from `Cold` (no prewarm happened) still works — it prewarms then runs, just slower. This matches architecture.md:588-590 ("measure NFR-5 with prewarm on AND off; if it shaves little, drop it") — so correctness must NOT depend on prewarm having run. The state machine degrades gracefully from "warm + instant" to "cold + slower" without changing the outcome.
4. **The camera-permission re-check (the inherited 3.1 deferral) reuses `CameraPermissionFlow.Evaluate()`, not a parallel permission source.** The 3.1 CR explicitly assigned Story 3.3 the re-validation of camera permission before session start / on resume (Android can revoke it while backgrounded). `ArSessionService` injects the registered `CameraPermissionFlow` and calls `Evaluate()` (live OS truth via `ICameraPermission`) before `StartAsync` and on backgrounded-resume; on not-granted it refuses + tears down + raises `OnArSessionInterrupted` so 3.1's `CameraPermissionView` re-grant flow takes over. 3.3 does NOT re-implement the disclose→prompt→denied machine — it only READS the resolved grant and refuses. One permission-truth source, no duplication. **Note the deliberate side effect (call out for the CR):** `CameraPermissionFlow.Evaluate()` is NOT a pure read — it SETS `CameraPermissionFlow.State` to `Granted`/`Disclosure` (see `CameraPermissionFlow.cs:62-68`) and returns it. So when `ArSessionService` calls it, it intentionally DRIVES the shared flow's state machine — flipping a stale `Denied`/`Granted` flow back to `Disclosure` on a revocation. This is the DESIRED re-routing ("route back to the disclosure/re-grant flow if revoked" — the exact 3.1-deferral instruction), not an accidental mutation; document it at the call site so the CR does not flag the cross-service state write as a surprise. `ArSessionService` reads the RETURNED `CameraPermissionState` (== `Granted`?) to decide; it does not itself mutate the flow.
5. **`ArcoreSession` device subsystem glue is a stub-and-defer (decision #5).** The AR Foundation `ARSession` is a scene `MonoBehaviour` and 3.3 builds no scene; the device subsystem start/stop is genuinely device-only + CI-untestable. So `ArcoreSession` ships its editor path real (the lifecycle is reachable + fully proven against the fake) and its device path as a documented `TODO(Story 3.4/6.3)` wired when the AR rig scene lands — exactly the "logic-complete-but-unplaced" disposition 3.1/3.2 views took. The lifecycle STATE MACHINE (the real value of 3.3) is fully tested; only the untestable subsystem glue waits.
6. **Bootstrap owns the `LoadPhase` staging contract, NOT the prewarm trigger.** Bootstrap defines `LoadPhase { EssentialSync, WarmupAsync, Ready }` and keeps AR off the awaited path (AC-CS-2). But it does NOT eagerly call `PrewarmAsync()` at boot — the architecture is explicit (architecture.md:583, :588-589) that Onboarding/Home own the prewarm trigger, gated to "AR-entry affordance visible/likely," because warming AR before the player is near AR burns battery/GPU. So 3.3's Bootstrap work is: the enum, the `ArSessionService` registration, and the staging shape (essentials sync / warmup-not-awaited) — the trigger itself is Epic 6's gated call into the view's `Prewarm()`.
7. **`OnArSessionInterrupted` is declared + raised here, consumed downstream.** The architecture names this event (architecture.md:650; AR-6). 3.3 raises it for the lifecycle-loss cases it owns (permission-revoked-on-resume; explicit loss teardown). The `EncounterStateMachine.Suspended` consumer + `TryRestoreAnchor` recovery are Story 3.5 / Epic 4. The event exists + fires now; its handler is downstream (recorded as a deferral).

### Edge cases & boundary rules (must-handle — these become test cases)
- **Cold → prewarm → ready:** `PrewarmAsync()` (supported+granted) → `Prewarming` (in-flight `StartAsync`) → `Ready` on completion; `StartCount == 1`.
- **Ready → enter → running:** `EnterAsync()` from `Ready` → `Running` (no second `StartAsync`).
- **Cold → enter (prewarm optional):** `EnterAsync()` from `Cold` (supported+granted) → prewarms then `Running`; `StartCount == 1` (prewarm is an optimization, not a precondition — decision #3).
- **Double-start safe (AC-3):** a second `PrewarmAsync()` while `StartAsync` is in-flight does NOT fire a second start (`StartCount == 1`); returns the in-flight task / warned no-op (decision #2).
- **Cheap tear-back (AC-3):** `Ready → TearDown()` → `Cold`, `Stop` called once. The back-out-before-committing path.
- **Backgrounding (AC-3):** `Running → OnApplicationPause(true)` → `Paused` + `Pause()` once; `OnApplicationPause(false)` (still granted) → `Running` + `Resume()` once.
- **Permission revoked while backgrounded (inherited 3.1 deferral):** `Running → pause → [permission revoked] → OnApplicationPause(false)` → does NOT `Resume()`; instead `TearDown()` → `Cold` AND `OnArSessionInterrupted` fires once (route back to re-grant). THE pin for the deferral 3.3 owns.
- **Unsupported device (NFR-3):** `!IsSupported` → `PrewarmAsync()` stays `Cold`, `StartCount == 0`, `GameLog.Warn`, no throw.
- **Permission not granted at prewarm (NFR-3):** supported but not granted → `PrewarmAsync()` stays `Cold`, `StartCount == 0`, warned, no throw.
- **Illegal transitions (NFR-3):** `TearDown()` from `Cold` → warned no-op; `OnApplicationPause(true)` from `Cold`/`Ready` → warned no-op; `EnterAsync` while `Running` → warned no-op, `StartCount` unchanged. None throw.
- **View inert (service unregistered / services not ready):** `Prewarm`/`EnterAr`/`BackOut`/the pump no-op + `GameLog.Warn`, never throw at `Awake` (the 2.4/2.5/3.1/3.2 graceful-degrade precedent).
- **AC-2 staging (the contract, not a duration):** `LoadPhase { EssentialSync, WarmupAsync, Ready }` exists in that order; `ArSessionService` does NOT warm (`StartAsync`) on construction/registration — warmup is on the async path, not awaited before Ready (the AC-CS-2 regression guard).
