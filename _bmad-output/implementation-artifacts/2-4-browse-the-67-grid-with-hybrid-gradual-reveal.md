# Story 2.4: Browse the 67-grid with hybrid gradual reveal

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a player,
I want a collection grid that shows what I've caught and hints at what's near without spoiling top tiers,
so that completing the Codex feels like a finite, escalating goal.

## Acceptance Criteria

**AC-1 — The grid renders 67 slots, each in one of three states, with the "X / 67" indicator** [Source: docs/epics.md#Story-2.4 (447-450); UX-DR5 epics.md:98; DESIGN.md:131-134; .decision-log.md:147-150 (OQ-7 hybrid)]
**Given** the Codex grid
**When** it renders the 67 slots
**Then** each slot shows one of three states: Discovered (full art + tier-tinted "caught" stamp), `???` silhouette (undiscovered in a tier the player has *begun* encountering), or `?` blank (undiscovered in a tier not yet reached)
**And** the visible "X / 67" progress indicator is shown.

**AC-2 — Climbing the Dread Ladder upgrades a newly-begun tier's blanks to `???`; higher unreached tiers stay `?`** [Source: docs/epics.md#Story-2.4 (452-456); EXPERIENCE.md:39-51 (rarity = Dread Ladder); .decision-log.md:151-154 (begun-tier principle)]
**Given** the player climbs the Dread Ladder (begins encountering a higher tier)
**When** that tier becomes "begun"
**Then** its previously-blank slots upgrade to `???` silhouettes ("The Veil shows you more…")
**And** higher unreached tiers remain `?` blank (preserving the reveal).

**AC-3 — A first discovery flips its slot silhouette→full art and the count ticks up** [Source: docs/epics.md#Story-2.4 (458-459); UX-DR5 epics.md:98; UX-DR15 epics.md:108; CodexService.OnMonsterDiscovered (Story 2.3)]
**Given** a first discovery resolves
**When** the Codex updates
**Then** the slot flips silhouette→full art with a tier-colored "caught" stamp and the count ticks up (e.g., "11 / 67 → 12 / 67").

> **Scope note on the visual verbs in AC-1/AC-3 (locked with the user — see the SCOPE section in Dev Notes).** "Full art", "tier-tinted caught stamp", "tier-colored", and the *animated* "flip"/"ticks up" are **Epic 6 visual concerns** (the Pumpkin-Patch tier-color tokens UX-DR2 + the chunky 3-state Codex-slot component UX-DR5 do not exist yet — they are Stories 6.1/6.2). This story delivers the **decision** behind each state (which of the three states each of the 67 slots is in, the begun-tier computation, the X/67 count, and the *recompute-on-discovery* that drives the flip + tick) as a headless-testable **presenter**, plus a thin logic-free `CodexGridView : MonoBehaviour` that binds it. The pixels — real art, tier colors, the flip animation, the count-tick animation, the chunky component — are explicitly **deferred to Epic 6** and recorded in `deferred-work.md`. The presenter exposes everything Epic 6 needs (per-slot state + tier + discovered-id-that-just-flipped + before/after counts) so the animation is a pure rendering concern when the component lands. **This is a deliberate partial-by-altitude split, not an unmet AC**: the *logic* of all three ACs is fully built and tested now; only the *rendering* of the already-decided states defers.

### The two design decisions this story locks (neither was specified in the docs — see References)
1. **Begun-tier trigger = discovery, "one tier ahead" (user-locked).** A tier T's undiscovered slots show as `???` silhouettes iff `(int)T <= floor + 1`, where `floor = max Rarity among discovered AUTHORED Monsters` (or `-1` if none). This is the only signal computable in Epic 2 — the data model tracks **discovery** (Capture/Slay set `SaveModel.Codex` keys, Story 2.3); "encounter without capture" (lure/scan) is **not tracked until Epic 4**.
   - **⚠ This rule SUPERSEDES the UX doc's literal proposal — flag for the CR reviewer so the difference is not mistaken for a bug.** `.decision-log.md:151-154` *proposes* begun(T) = "the player has encountered/**discovered their first monster of that tier**" (i.e. begun = ≥1 discovered IN tier T). That proposal was explicitly tagged a **balancing call to confirm at story detailing** (`.decision-log.md:154` "to confirm/feed upstream"; memory ux-feed-upstream-queue.md "residual tuning detail… decide during epic/story detailing"). The locked rule is the **one-tier-ahead** variant (begun = `T <= floor + 1`), which DIFFERS observably: under the doc's literal proposal a brand-new player sees **all 67 blank** (no tier has a discovery yet); under the locked rule a brand-new player sees authored **Common slots as `???`** (one tier ahead of "nothing"). This is a sanctioned design decision (user-locked, 2026-06-13), NOT an implementation of the doc's wording. **Epic 4 may widen "begun"** to include lure/scan encounters once that signal exists — `IsTierBegun` is the single seam to revisit.
2. **Reserved/unauthored slots = `?` Blank always (user-locked).** 62 of 67 ids have no `MonsterDefinition` → no Rarity → cannot be tiered → cannot be silhouetted. They render `?` Blank until content authoring assigns a definition. No invented placeholder rarities (that canon belongs to the content/art track). The grid grows silhouettes organically as monsters are authored. **Exception: a *discovered* unauthored id still shows Discovered** (discovery is the strongest signal and is tracked independently of authoring); its `Tier` is `null` and the view falls back (Epic 6's concern).
