# Story 2.3: Read and persist Codex collection state

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a player,
I want my discovered Monsters tracked persistently,
so that my collection is remembered and progress counts correctly.

## Acceptance Criteria

**AC-1 — First discovery (via Capture or Slay) creates/updates the Codex entry, persists it, raises `OnMonsterDiscovered`, and increments "X / 67" by exactly one** [Source: docs/epics.md#Story-2.3 (422-425); docs/architecture.md#Codex-idempotency (661-662); AR-6 epics.md:63; AR-8 epics.md:65]
**Given** `CodexService` (folded in Monsters) as a read model over owned-monster state + the catalog
**When** a Monster is first discovered (via Capture or Slay)
**Then** its Codex entry is created/updated in `SaveModel` and `OnMonsterDiscovered` is raised
**And** discovering a new Monster increments the "X / 67" count by exactly one.

**AC-2 — Re-discovery of an already-owned Monster is idempotent (count does NOT increment)** [Source: docs/epics.md#Story-2.3 (427-429); docs/architecture.md#Codex-idempotency (661)]
**Given** an already-owned Monster
**When** it is captured/slain again (idempotency)
**Then** the "X / 67" count does NOT increment.

**AC-3 — Reaching 67/67 raises `OnCodexCompleted`** [Source: docs/epics.md#Story-2.3 (431-433); docs/architecture.md#Codex-idempotency (662)]
**Given** the collection reaches 67/67
**When** the final Monster is discovered
**Then** `OnCodexCompleted` is raised.

**AC-4 — Codex state survives an app restart (persisted entries + progress count preserved)** [Source: docs/epics.md#Story-2.3 (435-437); FR-12 epics.md:35; docs/architecture.md#Save-contents (186)]
**Given** persisted Codex state
**When** the app is closed and reopened
**Then** all Codex entries and the progress count are preserved.

If `model.Codex.ContainsKey(id)` AND the requested `via` flag is already set, this is a **no-op**: return `CodexResult.AlreadyRecorded(DiscoveredCount)` WITHOUT a persist and WITHOUT raising `OnMonsterDiscovered`. **Subtlety:** a Monster discovered via Capture and later Slain is "already discovered" (no count increment, no `OnMonsterDiscovered`) but the `Slain` flag IS newly set → that is a legitimate persist (the entry changed) but NOT a discovery event and NOT a count increment. ### What this story IS and IS NOT
- **IS:** `CodexService` (in `Monsters/Codex/`, assembly + namespace `Veilwalkers.Monsters`) — the **read model** over `SaveModel.Codex` + `MonsterDatabase` (`DiscoveredCount`, `IsDiscovered`, `GetEntry`, `UniverseCount`), the two AR-6 events (`OnMonsterDiscovered`, `OnCodexCompleted`), and ONE atomic write seam `RecordDiscoveryAsync(id, DiscoverySource)` that creates/updates the entry, persists once under CodexService's **own** `SemaphoreSlim` gate, and raises the events with idempotency + 67/67 completion. Plus the `Monsters → Persistence` asmdef edge (and its acyclic-matrix update). Persistence across restart is verified (the field already exists; 2.3 proves the write/read round-trips).
- **IS NOT:** the Codex grid UI / 3-state hybrid reveal (**Story 2.4**), the detail view (**Story 2.5**), any new `SaveModel` schema/field (the `Codex` dict + `CodexEntryData` already exist from Epic 1, schema v2 — **no schema change, no migration**). NOT the Capture/Slay Encounter actions or the composed multi-delta action write (**Epic 4** — 2.3 ships the standalone composable primitive only). NOT `RecordScanAsync`/scan-flag semantics or variant-flag writing (Epic 4 / later content). Do NOT touch Credit/XP/charge state — CodexService is read-model-only w.r.t. economy.

### The decision-of-record for this story (2 design choices already made)
1. **Write surface = "discovery-record seam":** one `RecordDiscoveryAsync(id, DiscoverySource via)`, NOT per-action `RecordCaptureAsync`/`RecordSlayAsync`/`RecordScanAsync`. Rationale: Capture & Slay (Epic 4) both funnel through one seam; per-action method proliferation pre-commits flag semantics (does Slay imply Captured? scan-before-discovery?) that are Encounter concerns. 2.3 sets only the single flag for the `via` it's told.
2. **Atomic-write = standalone now, composable later:** `RecordDiscoveryAsync` does its OWN single atomic persist under CodexService's **own private `SemaphoreSlim`** (self-contained, correct standalone — mirrors `DailyRewardService`'s shape, minus the cross-service lock sharing). Epic 4 composes the full Capture-action write (charge spend delta + codex delta → one `SaveAsync`) later. 2.3 documents that seam but builds no multi-delta orchestrator.
   - **Why a private lock, NOT the shared `SaveMutationLock` (VS-resolved):** the Economy `SaveMutationLock` lives in the `Veilwalkers.Economy` assembly; `CodexService` is in the sibling `Veilwalkers.Monsters` assembly, so referencing it would force an **illegal `Monsters → Economy` sideways edge** (AR-5 siblings; `AcyclicDependencyTests` rejects it). The shared lock's job — stop two *different* services from durably persisting each other's uncommitted delta on the same `SaveModel` — does NOT apply in 2.3: CodexService is the lone codex writer, and no single action mutates both economy state AND codex until **Epic 4's Capture/Slay**. That composed write lives in `Veilwalkers.Encounter` (which legally references BOTH Economy and Monsters), and Encounter serializes it under the shared Economy lock THERE. Gameplay is single-threaded, so a 2.3 codex write and a concurrent Economy write cannot actually interleave today. AR-8 ("one atomic write per action") is fully honored by CodexService's self-contained locked single-`SaveAsync` write.

### Architecture constraints (hard rules)
- **`CodexService` lives in `Monsters/Codex/`, assembly `Veilwalkers.Monsters`** — Codex folds into Monsters; there is NO `Veilwalkers.Codex` assembly despite architecture.md:289 listing it in one inventory line (line 398 + 441 are authoritative: "Codex folded into Monsters; Pit has no asmdef"). - **Codex idempotency (AR canon):** re-discovering an owned Monster does NOT increment "X/67"; only first discovery counts. 67/67 raises `OnCodexCompleted`. [docs/architecture.md:661-662]
- **Events are the UI seam (AR-6):** `OnMonsterDiscovered`, `OnCodexCompleted` named `On<Subject><PastTense>`. UI (Epic 2.4/Epic 6) subscribes in `OnEnable` / unsubscribes in `OnDisable` (symmetric) — but the UI is NOT in this story; 2.3 only declares + raises the events. [AR-6 epics.md:63; architecture.md:278]

### The house pattern to copy verbatim (DailyRewardService is the closest twin)
`RecordDiscoveryAsync` is structurally identical to `DailyRewardService.TryClaimAsync` (DailyRewardService.cs:78-174) — a guarded, locked, single-persist, rollback-safe, event-after-release mutator. Copy that skeleton:
1. Validate input → throw for programmer error (invalid id), like `CreditService` throws on `cost<=0`.
2. `await _gate.WaitAsync().ConfigureAwait(false)` → `try { … } finally { _gate.Release(); }` — where `_gate` is CodexService's OWN `private readonly SemaphoreSlim _gate = new SemaphoreSlim(1, 1);` (NOT the Economy `SaveMutationLock`; see decision-of-record #2).
3. `SaveModel model = RequireModel();` (snapshot the ref ONCE; add a private `RequireModel()` that throws `InvalidOperationException` if `_saveService.Current == null`, mirroring the Economy services).
4. Idempotency/no-op short-circuit (return without persist).
5. Capture rollback state (key-pre-existed?, prior-flag) → mutate in memory.
6. `await _saveService.SaveAsync().ConfigureAwait(false);` (one write).
7. Recovery-swap guard: `if (ReferenceEquals(_saveService.Current, model))` commit; else roll back + `PersistenceFailed`.
8. `catch` → roll back + `PersistenceFailed`.
9. After `finally`/lock-release: raise `OnMonsterDiscovered` (if committed first-discovery), then `OnCodexCompleted` (if that made it 67), each in try/catch + `GameLog.Error`.

**Event ordering is NOT negotiable:** events fire AFTER the lock releases and ONLY on a committed mutation — never inside the lock, never on a rolled-back write (CreditService.cs:136-143, DailyRewardService.cs:168-170). A subscriber that re-enters the service must not deadlock on the held lock — hence raise-after-release.

### Edge cases & boundary rules (must-handle — these become test cases)
- **First discovery vs. re-discovery vs. flag-flip:** key-absent → create+persist+event+increment. Key-present & via-flag-already-set → pure no-op (no persist, no event, no increment). Key-present & via-flag-NOT-set (e.g. captured then slain) → persist (entry changed) but NO event, NO increment. (Task 2 idempotency subtlety; Task 5 `Capture_then_slay…` test.)
- **67/67 exactly once:** `OnCodexCompleted` fires only on the `isFirstDiscovery && newCount == UniverseCount` transition. Never on re-discovery of an owned monster after 67/67 (would double-fire). [architecture.md:661-662]
- **Invalid / out-of-universe id:** `mon00`, `mon68`, `xyz`, null, "" → `ArgumentException` BEFORE any mutation/persist (programmer-error contract). A valid-but-unpopulated id (e.g. `mon40`, not authored in MVP) is still a legal discoverable universe id — accept it. [MonsterDatabase.IsValidMonsterId]
- **Persist fault / recovery swap:** roll back the exact in-memory mutation (remove a newly-created entry; restore a flipped flag) on the captured `model` ref, return `PersistenceFailed`, raise no event. [CreditService.cs:100-119 swap pattern]
- **Throwing event subscriber:** swallowed + `GameLog.Error`; the committed write is not faulted. [CreditService.cs:222-233]
- **Reentrancy:** events raised after lock release, so a subscriber may safely call back into a read method (idempotent) without deadlock.
