# Story 1.9: Reserve the ad-reward and telemetry seams (stubs)

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a developer,
I want the OQ-4 ad-reward hook and local telemetry seams reserved now,
So that a future ad grant and balancing pass do not retro-break the economy.

> **This is a SEAM-RESERVATION story.** The deliverable is that the seams EXIST, are WIRED, and are TESTED to honor their contracts — NOT that they are called from gameplay. Ads are not built (OQ-4); the telemetry `Count(...)` call sites on Capture/Slay/zero-credit belong to the epics that own those actions (Epic 4, and the zero-credit moment to whoever first detects it). 1.9 builds the reusable machinery so those later call sites have a defined, non-economy-breaking target.

## Acceptance Criteria

### Ad-reward seam (AR-17)

**AC-AD-1 — `AdHook.GrantReward()` grants exactly one low-tier charge via Progression, never Credits/XP**
**Given** the ad-reward seam `App/AdHook`
**When** `GrantReward()` is invoked (stub; ads not built)
**Then** it calls `IProgressionService.AddChargeAsync(...)` with a low-tier charge (`ChargeType.StabilityBoost`) and grants exactly one charge
**And** it never calls `CreditService` and never touches raw XP — the only mutation is the one charge-count bump `AddChargeAsync` performs.

> **Which charge — read carefully (the architecture is explicit and counterintuitive).** `architecture.md:547-549`: "an ad grants the **weakest / a randomized low-tier charge** … otherwise an on-demand **Strong Capture** charge defeats the earned-progression curve." So **`StrongCapture` is the charge an ad must NOT hand out** — it is the progression-sensitive one, named as the anti-example. Do NOT pick `StrongCapture` just because it is `ChargeType` value `0`; the enum order is a serialization-stability order, NOT a power order (its doc-comment calls Strong Capture "a *higher-tier* capture attempt"). This story pins a DETERMINISTIC low-tier charge — `StabilityBoost` — for the stub; the architecture's "randomized low-tier" and the exact tier are an OQ-4 tuning decision (would need a seeded-RNG seam to stay testable). `StabilityBoost` (an encounter-steadying aid) is a low-tier choice that does not defeat the capture curve. [Source: docs/architecture.md#AR-17 §"Which charge (refined)" (lines 547-549); Assets/Veilwalkers/Economy/ChargeType.cs:17 (StrongCapture = "higher-tier")]

[Source: docs/epics.md#Story 1.9; docs/architecture.md#AR-17, §"OQ-4 ad-reward hook" (lines 518-519, 547-556); Assets/Veilwalkers/Economy/IProgressionService.cs:85-92 (AddChargeAsync is the 1.9 ad-reward seam, one charge per call)]

**AC-AD-2 — A daily cap is enforced via `SaveModel.AdReward {grantsToday, dayBucket}` keyed on the `IClock` day-bucket integer**
**Given** `EconomyConfig.AdDailyCap` grants/day and `SaveModel.AdReward {GrantsToday, DayBucket}`
**When** `GrantReward()` is invoked
**Then** if the stored `DayBucket` is NOT the current UTC day-bucket, the counter resets (`GrantsToday = 0`, `DayBucket = today`) before the cap check (a new day re-opens the cap)
**And** if `GrantsToday >= AdDailyCap` for the current day-bucket, the grant is refused (no charge, no counter change) and a typed result reports the cap was hit
**And** on a granted reward, `GrantsToday` increments and the `AdReward` counter is persisted
**And** the day-bucket is a `long` integer derived ONLY from `IClock.UtcNow` (never a `DateTime`, never `DateTime.Now`).
[Source: docs/epics.md#Story 1.9; docs/architecture.md#AR-17 (lines 553-554: "store the IClock-derived day-bucket integer, never a DateTime"), AR-18; Assets/Veilwalkers/Persistence/AdRewardData.cs; Assets/Veilwalkers/Economy/EconomyConfig.cs:113 (AdDailyCap)]

**AC-AD-3 — The ad grant follows App→Economy direction; the cap counter writes atomically and the charge is the Progression atomic write**
**Given** `AdHook` lives in App and `ProgressionService` in Economy
**When** a reward is granted
**Then** the call direction is App→Economy (downstream→upstream, no cycle) and the charge is granted through `AddChargeAsync`'s existing lock-held atomic mutate→persist→rollback pipeline
**And** the `AdReward` cap-counter write is itself committed (it is a `SaveModel` mutation; persist it under the same shared `SaveMutationLock` discipline, OR — see Dev Notes deviation #2 — fold the counter into the same atomic step so a granted charge and its cap-increment cannot diverge).
[Source: docs/architecture.md#AR-17 (line 551-552 App→Economy), AR-8 (atomic write per action); Assets/Veilwalkers/Economy/ProgressionService.cs:214-281 (the AddChargeAsync atomic pattern)]

**Given** a test double of the progression/credit surface
**When** `GrantReward()` is invoked
**Then** a spy asserts the ONLY economy mutation is the charge grant — `CreditService` is never called and the XP value is never changed.

### Telemetry seam (AR-15)

**AC-TEL-1 — `ITelemetrySink.Count(evt, amount)` in Core, `LocalTelemetrySink` in App, swappable for Firebase**
**Given** the telemetry seam
**When** `ITelemetrySink.Count(evt, amount)` is called (`LocalTelemetrySink` is the App impl)
**Then** the call is recorded as a counter increment for `evt` by `amount`
**And** `ITelemetrySink` is a pure Core interface with no impl detail leaking, so a Phase-2 Firebase sink replaces `LocalTelemetrySink` WITHOUT moving any call site.
[Source: docs/epics.md#Story 1.9; docs/architecture.md#AR-15 (lines 570-576: "Seam ITelemetrySink.Count(evt, amount) (Core); LocalTelemetrySink impl in App … swappable")]

**AC-TEL-2 — Counters persist to a SEPARATE `telemetry.json` via `ITelemetryStore` (capped ring), never to `SaveModel`**
**Given** `ITelemetryStore` (Core seam) + `LocalTelemetryStore` (Persistence impl)
**When** telemetry counters are written
**Then** they go to a separate `telemetry.json` in `Application.persistentDataPath` (path injected, atomic temp+swap, mirroring `LocalProgressStore`), NEVER into `SaveModel`
**And** the store is a CAPPED RING — total retained entries are bounded (by count and/or `EconomyConfig.TelemetryRetentionDays`) so per-action/per-session counters cannot grow unbounded and bloat the save/migration surface
**And** no telemetry field is added to `SaveModel`.
[Source: docs/epics.md#Story 1.9; docs/architecture.md#AR-15 (lines 571-576: "Storage is a SEPARATE telemetry.json via ITelemetryStore (capped ring), NOT SaveModel"); Assets/Veilwalkers/Persistence/LocalProgressStore.cs:46-100,203-230 (the atomic-write + path-injection pattern to mirror)]

**AC-TEL-3 — The bounded write-once `firstZeroCreditDay` scalar lives in `SaveModel`; `EconomyConfig` holds tunables, never counters**
**Given** the architecture's explicit exception (a bounded, write-once scalar MAY live in `SaveModel`)
**When** the first-zero-credit day is recorded
**Then** `SaveModel.FirstZeroCreditDay` (existing `long`, default `-1` = unset) is set ONCE — write-once: a later zero-credit moment does NOT overwrite a set value
**And** the tunables `adDailyCap`/`telemetryRetentionDays` are read from `EconomyConfig` (they already exist), and NO counter ever lives in `EconomyConfig`.
[Source: docs/epics.md#Story 1.9; docs/architecture.md#AR-15 (lines 573-575: "the bounded, write-once firstZeroCreditDay scalar MAY live in SaveModel. EconomyConfig holds tunables … never the counters"); Assets/Veilwalkers/Persistence/SaveModel.cs:90-95 (FirstZeroCreditDay long, -1)]

> **AC scope note.** AC-TEL-3's "record the first-zero-credit day" is the ONE concrete telemetry write 1.9 owns the mechanism for (a small `MarkFirstZeroCreditDay(...)`-style helper that is write-once). The broader instrument (loop-completions-before-zero, session-after-first-zero) is the *call-site* work of later epics; 1.9 provides the sink/store they will call, not those call sites. Do not instrument Capture/Slay here.

### Ad-reward seam

- [x] **Task 5 — `AdHook` in App + the day-bucket-long helper (AC: AD-1, AD-2, AD-3)**
  - [x] `Assets/Veilwalkers/App/AdHook.cs` (NEW, stub) — ctor-injects `IProgressionService`, `SaveService`, `SaveMutationLock`, `IClock`, and the `int adDailyCap` value (inject the int, NOT the whole `EconomyConfig` — Bootstrap reads `_economyConfig.AdDailyCap` and passes it, matching how `LocalTelemetrySink` takes the retention int; null-guard the reference args, and treat `adDailyCap <= 0` as a misconfiguration → throw in the ctor, like `DailyRewardService` does for its amount). Returns the typed `AdRewardResult` (Task 5c). Public `Task<AdRewardResult> GrantRewardAsync()`:

    **⚠ Lock discipline — this is the load-bearing correctness point.** `SaveMutationLock` is a NON-reentrant `SemaphoreSlim(1,1)` (`SaveMutationLock.cs:31`), and `AddChargeAsync` acquires it INTERNALLY (`ProgressionService.cs:224`). Therefore AdHook must NOT hold the lock across the `AddChargeAsync` call — doing so self-deadlocks. The flow drops the lock between the cap check and the charge, and re-acquires it for the counter write:

    1. `long todayBucket = DayBucket(clock.UtcNow);` (Task 5b).
    2. **Acquire `SaveMutationLock`** (`WaitAsync`/`finally Release`). Read `SaveModel.AdReward`. If `AdReward.DayBucket != todayBucket`, roll the day over (`GrantsToday = 0; DayBucket = todayBucket`). If `AdReward.GrantsToday >= adDailyCap`, **release the lock and return `AdRewardResult.CapReached(...)`** (no charge, no persist). Otherwise **release the lock** — do NOT keep holding it. (The rollover here is in-memory only; it does not need its own persist — the next granted write, step 4, persists it, and if no grant happens the rollover is recomputed next call.)
    3. `await _progression.AddChargeAsync(StabilityBoost)` — this acquires/releases the lock ITSELF (which is why step 2 released first). On a FAILED `Result`, return `AdRewardResult.Failed(...)` — the cap counter is untouched, so the grant simply retries next time (no cap consumed for a charge that didn't land).
    4. On charge success: **re-acquire `SaveMutationLock`**. Re-read `AdReward` and re-apply the rollover check (the lock was dropped, so a concurrent mutator could have changed it — TOCTOU; re-check `DayBucket` against `todayBucket` and reset if needed). Increment `GrantsToday`. `await _saveService.SaveAsync()`. Release. Return `AdRewardResult.Granted(newGrantsToday)`.

    **Atomicity worst case (document in the method, per 1.7/1.8 precedent):** the charge (step 3) and the counter increment (step 4) are TWO separate persists — they cannot be one atomic write without editing `ProgressionService` (out of scope). Ordering is charge-FIRST so the benign failure direction holds: a crash between step 3 and step 4 leaves the charge granted but `GrantsToday` un-incremented → at most ONE extra ad-charge that day (a benign over-grant). It can NEVER consume the cap without granting a charge, and can never let the player permanently exceed the cap. This mirrors 1.7's grant-then-marker reasoning; for a non-wired single-player stub the TOCTOU is acceptable, and step 4's re-check is the guard.
    - **`GrantReward()` (epic wording) vs `GrantRewardAsync()`:** async is mandatory (it persists), so the real method is `GrantRewardAsync()` — the same documented `Async`-suffix deviation `CreditService`/`DailyRewardService`/`AddChargeAsync` all carry. No sync shim is needed (no caller exists yet; the stub is invoked by tests and, later, OQ-4's ad SDK). [Source: Assets/Veilwalkers/Economy/SaveMutationLock.cs:31 (non-reentrant); Assets/Veilwalkers/Economy/ProgressionService.cs:224 (AddChargeAsync takes the lock); Assets/Veilwalkers/Economy/IProgressionService.cs:85-92; Assets/Veilwalkers/Economy/DailyRewardService.cs (inject-value + lock-held pattern); Assets/Veilwalkers/App/Bootstrap.cs:124-125]

  - [x] **Task 5b — the day-bucket-long algorithm.** 1.9 is the FIRST to derive a day-bucket `long` from `IClock` (Story 1.8's `DailyRewardService` used a `yyyy-MM-dd` STRING — a DIFFERENT representation, intentionally: `DailyClaim` is a human-readable date, the day-bucket is a compact integer for `AdReward.DayBucket`/`FirstZeroCreditDay`). Define it ONCE, reused by both the ad-cap (Task 5) and the first-zero recorder (Task 4), so they agree. Recommended: `long DayBucket(DateTime utc) => utc.Date.Ticks / TimeSpan.TicksPerDay;` (the integer count of whole days since `DateTime.MinValue`; `.Date` zeroes the time-of-day so all instants in a UTC day map to one bucket; `.Ticks` is `Kind`-independent so a `FakeClock`'s `Unspecified`-kind instant works the same as a real UTC one). `DateTime.UnixEpoch` exists in Unity 2022's .NET Standard 2.1 too, but `Ticks / TimeSpan.TicksPerDay` is the simpler explicit day-index and needs no epoch constant — use it. **Put this helper in ONE place both Task 4 (first-zero recorder) and Task 5 (AdHook) reach** — a single App-layer static (e.g. `App/DayBucket.cs` `internal static long For(DateTime utc)`) is correct, since both callers are App-layer; do NOT duplicate the formula in two files (a reviewer will flag drift). If a non-App caller ever needs it, promote to Core then. ### Sanctioned-deviation matrix (check before classifying any review finding)

1. **This is a STUB/seam story — the telemetry `Count(...)` call sites and the ad `GrantReward()` trigger are NOT wired into gameplay.** The deliverable is the machinery + its contract tests, like 1.7/1.8 delivered grant/claim LOGIC without UI. A reviewer asking "where is Capture calling `Count`?" or "what triggers `GrantReward`?" is asking for Epic 4 / OQ-4-build work that this story explicitly reserves-but-does-not-build (see the story's scope note + epic title "stubs"). The ONE concrete write 1.9 owns is the write-once `firstZeroCreditDay` recorder.
2. **The ad-cap counter (`AdReward.GrantsToday`) and the granted charge should not be able to diverge.** Two valid designs: (a) two lock-held writes (charge via `AddChargeAsync`, then the counter), ordered charge-first so a crash between them leaves `GrantsToday` un-incremented → at most the player gets ONE extra ad grant that day (a benign over-grant, never a charge-without-cap-consumed that would let them exceed the cap) — analyze and pick the benign direction; OR (b) fold the counter increment into the same atomic write as the charge (read `AddChargeAsync`'s internals would require touching Economy, so (a) is more likely). Whichever is chosen, the dev MUST document the crash-failure worst case explicitly (as 1.7's grant-then-marker and 1.8's single-write notes did). A reviewer flagging "the cap and the charge are two persists" is correct to ask, but the ordered-benign-direction design ANSWERS it — it is not an unhandled gap if the ordering is justified.
3. **The ad reward is a DETERMINISTIC low-tier charge (`StabilityBoost`), not randomized — and explicitly NOT `StrongCapture`.** The architecture (`architecture.md:547-549`) names `StrongCapture` as the charge an ad must NOT grant ("otherwise an on-demand Strong Capture charge defeats the earned-progression curve") and asks for "the weakest / a randomized low-tier charge." `ChargeType`'s enum order is serialization-stability order, not power order — `StrongCapture = 0` is the STRONGEST, not the weakest, so picking it by ordinal would be exactly backwards. The stub pins `StabilityBoost` (a low-tier encounter aid) deterministically; randomizing across low tiers (and the precise tier) is OQ-4 tuning needing a seeded-RNG seam. A reviewer wanting randomization, or wanting `StrongCapture` "because it's value 0," is contradicting AR-17.
4. **`telemetry.json` is PLAINTEXT, not AES-encrypted.** The save file is encrypted (`LocalProgressStore` → `AesCrypto`) because it holds tamper-sensitive currency/balance state; telemetry holds only diagnostic counters + event keys (no PII, no currency, no anti-cheat-sensitive state), is a LOCAL diagnostic for testers to inspect, and the architecture specifies encryption for the SAVE (line 300) but not telemetry (lines 569-576). Plaintext is intentional and is asserted by a test (the file deserializes as JSON without decryption). **Corollary the call sites must honor:** because telemetry is plaintext, nothing sensitive may ever be put INTO it — keep it to counters and stable event keys (the day-bucket is fine; never log balances, ids, or anything that's encrypted in the save). A reviewer flagging "telemetry isn't encrypted like the save" is contradicting the diagnostic-inspection intent; if a later phase wants it encrypted, `AesCrypto` is available.
5. **Telemetry counters live in a SEPARATE `telemetry.json`, never in `SaveModel`; the ONLY SaveModel scalar 1.9 touches is the pre-existing write-once `firstZeroCreditDay`.** This is the explicit AR-15 rule ("NOT SaveModel … Exception: the bounded write-once firstZeroCreditDay MAY live in SaveModel"). A reviewer wanting the counters in SaveModel is contradicting AR-15; a reviewer flagging `firstZeroCreditDay` in SaveModel as a violation has it backwards (it is the sanctioned exception — bounded + write-once).
6. **`LocalTelemetryStore` (Persistence) does NOT read `EconomyConfig` for the retention cap.** Persistence does not reference Economy (not in the allowed-edge matrix). The cap value is read from `EconomyConfig.TelemetryRetentionDays` at the App/Bootstrap composition boundary and INJECTED into the store/sink as a plain int. A reviewer wanting the store to read `EconomyConfig` directly is asking for an illegal Persistence→Economy edge.
7. **`AdHook`/`LocalTelemetrySink` ctor-inject their deps (AR-4); they don't read `GameServices.Get<T>()`.** App composes them in Bootstrap. The day-bucket helper is the only shared logic; it is a pure static (no locator).
